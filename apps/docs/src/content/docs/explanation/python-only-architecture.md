# Why a Python-Only Prompt Runtime

Why does every slash command, every prompt template, and every task execution step in DeviaTDD live inside the Python package — never as a `.sh` wrapper, never as a separate file at the repo root, never as a binary on the operator's `$PATH`? The answer is a small cluster of decisions that, taken together, push the entire prompt layer through the same version-pinned, import-resolved, tool-locked machinery as the rest of the framework's source. The trade-offs that cluster makes are about reproducibility, blast radius, and the cost of admitting a new kind of file into a system whose other guarantees were already designed around Python.

## Context

The first version of this codebase carried the slash-command library in a separate top-level `prompts/` directory, in `Markdown` files alongside hand-rolled `<SKILL_DIR>/deviate-*.sh` invocation wrappers. The shape was convenient for a hackathon: each command was a Markdown file with a sibling shell entrypoint, and the agent platforms discovered commands by reading the directory. The shape was also fragile. Each `.sh` wrapper had to be kept in sync with its Markdown counterpart by hand, the wrappers depended on a `deviate` binary that might or might not be on `$PATH` at the moment the agent ran, and the contents of `prompts/` were not version-pinned against the contents of `src/` — a contributor who bumped `pyproject.toml` and a contributor who edited a `.md` file did not see their changes as the same release. ISS-001-005 ("CLI Architecture Realignment & Skill Integration") ended that shape by deleting every `.sh` file under `prompts/`, moving the Markdown sources into the Python package at `src/deviate/prompts/commands/<name>.md`, and routing every dispatch through a single loader in `src/deviate/prompts/assembly.py`.

The constraint that pushed the redesign was not aesthetic. It was that the constitution's Definition of Done includes "all task execution runs through `uv run`" — meaning every agent-visible action resolves to a process whose Python interpreter is project-locked by `mise.toml`'s `[env] python = "3.13"` block. A `.sh` wrapper that exec's `python -m deviate.cli …` and a Python function that runs `deviate.cli …` in-process look superficially similar, but only the second one inherits the `uv`-managed virtualenv, the locked `pyproject.toml` dependency set, the editable-install link to the source tree, and the in-process `_PHASE_MAP` dispatch with no subprocess round-trip. The shell wrapper forfeits all four. Re-architecting around "no shell wrappers" was the smallest change that preserved them.

The same logic extends to the agent-platform install mirror pattern. The agent platforms the project supports — Claude Code, OpenCode, Factory Droid, Pi coding agent — each discover their own slash-command set by reading a specific directory (`.claude/commands/`, `.opencode/commands/`, `.factory/commands/`, `.pi/prompts/` respectively). DeviaTDD installs command templates into all four directories during `deviate setup`, but the install is one-way: the package resource at `src/deviate/prompts/commands/<name>.md` is the source of truth, and the four mirrors are generated by `_install_commands_to_agents`. Editing a mirrored file in `.opencode/commands/` does not propagate back; the next `deviate setup` will overwrite it from the canonical source. The symmetric pairing extends further: an operator's global OpenCode skills directory at `~/.config/opencode/skills/` is treated as a read-only install mirror in the same sense — it is an output target, not a source of truth.

## Rationale

Three decisions stack on top of each other, and each one enables the next.

The first decision is *the prompt template is a package resource*. A Markdown file at `src/deviate/prompts/commands/deviate-red.md` is loaded at runtime through `importlib.resources.files("deviate.prompts.commands")` — the same mechanism that loads the package's own code, the same edit-time tracking that catches uncommitted changes, the same version-pinning that holds when the agent runs in a clean worktree. There is no separate prompt-version field, no prompt-hash drift, no opportunity for one contributor to ship a `pyproject.toml` bump without bumping the matching prompt. `deviate`'s `load_template()` in `src/deviate/prompts/assembly.py` is the single point of resolution: it locates the resource, compares its mtime against the calling Python process's sys.path entry, and re-reads it when those diverge. A prompt that is not a package resource cannot be located this way, because the project has no other handle on its version.

The second decision is *layer routing is a single dict*. The `_LAYER_MAP` constant in `assembly.py` maps every template name (`"explore"`, `"research"`, `"prd"`, `"shard"`, `"specify"`, `"plan"`, `"tasks"`, `"red"`, `"green"`, `"refactor"`, `"yellow"`, `"judge"`, `"execute"`, plus the implicit `None` for templates with no preamble) to a layer prefix (`"macro-auto"`, `"meso-auto"`, `"micro-auto"`). `load_template()` consults the dict once per invocation, fetches the corresponding layer preamble from `deviate.prompts.core.{layer}.md`, and concatenates it with the phase-specific template from `deviate.prompts.auto.{template_name}.md`. The result is a single string the agent sees as the assembled prompt — but the layer preamble is shared across every phase in that layer, the phase template is unique to one phase, and the boundary between them is enforced by the dict, not by convention. A template that is loaded by a `.sh` wrapper cannot consult this dict, because the wrapper is a separate process with no Python state to share; the wrapper would re-implement the routing by hand and would inevitably diverge.

The third decision is *every `mise run` invocation flows through `uv run`*. The `[tasks.<name>]` entries in `mise.toml` (`uv run pytest`, `uv run ruff check`, `uv run ruff format`, `uv run deviate`, the install task, the run-loop dev entrypoint) all share the same prefix. The reason this matters is that `uv run` resolves the project's virtualenv at invocation time, honors the `pyproject.toml` lockfile, and ensures the editable install is in scope; a bare `pytest` or `python` resolved against the operator's `$PATH` might point at a different Python, with a different `deviate` version, in a different virtualenv. The tool-locked form is the only one that fails closed when the toolchain is in a bad state; the unlocked forms fail open in ways that are not instrumented. `mise` is the dispatch surface (`mise run <task>`), `uv run` is the subprocess shape inside every dispatch, and the README's quick-start inherits both.

```
_LAYER_MAP (excerpt, src/deviate/prompts/assembly.py)
  "explore"/"research"/"prd"/"shard"  →  "macro-auto"
  "plan"/"tasks"                      →  "meso-auto"
  "red"/"green"/"refactor"            →  "micro-auto"
```

## Mental Model

Picture the entire prompt layer as a Python module the package carries with itself, the same way it carries its own modules. The agent does not "look up" a slash command; the agent asks `deviate` to assemble a prompt, and `deviate` resolves the request to a set of `importlib.resources.files()` calls, concatenates the result, and hands the string to whichever backend (`opencode`, `claude`, `factory`, `pi`) was selected at `deviate setup` time. The agent's view of the command — its name, its description, the autocomplete entry it shows — is a *mirror* of the package resource: the mirror is generated during `deviate setup` by `_install_commands_to_agents`, lives at `<workdir>/.<agent>/commands/<name>.md` (or `<workdir>/.pi/prompts/<name>.md` for Pi, which uses a different directory convention), and is overwritten on each `setup` run. The agent reads from the mirror; the package writes from the source.

```
                          PACKAGE source-of-truth
   src/deviate/prompts/commands/deviate-red.md
                       │  importlib.resources.files()
                       ▼
   src/deviate/prompts/assembly.py::load_template()
   │  (_LAYER_MAP lookup → layer preamble + auto/{template}.md)
   │
   ▼  joined prompt string
   │
   ├──►  in-process dispatch (CLI pre/post, _PHASE_MAP)
   │         │
   │         ▼
   │       stdout
   │
   └──►  agent backend (opencode / claude / factory / pi)
             │
             ▼
       <workdir>/.<agent>/commands/deviate-red.md   (mirror;
         regenerated each `deviate setup`; never edited directly)
```

The mirror is one-way by construction. `_install_commands_to_agents` compares the file content against the package-resource source on every install; an edit made at the mirror side is overwritten by the next install, and the source-of-truth invariant is preserved. The same one-way property extends to the global agent skills directory at `~/.config/opencode/skills/` (and its analogues for the other agents): the directory is a *target* of an install operation, not a source. DeviaTDD never reads from it; an operator who wants to customize a command edits the corresponding file under `src/deviate/prompts/commands/`, not the mirror, because the mirror is allowed to be overwritten on any future `deviate setup` run.

## Trade-Offs

What was gained. The python-only architecture earns four properties the previous shell-wrapper shape could not. First, prompt versioning is unified: there is one `pyproject.toml`, one lockfile, one editable install, and every prompt the agent sees is version-pinned against the same release as the code that loads it. Second, dispatch is in-process when it can be (the `_PHASE_MAP` synchronous function calls inside `deviate run`), and shell-out only when the agent platform demands it (the `subprocess.run` to `pi`, `opencode`, `claude`, or `factory`); there is no third tier of dispatch. Third, the `_LAYER_MAP` is a single-source-of-truth for layer membership, auditable in 14 lines of code, and any future layer (say, `product-auto`) is added by extending that one dict. Fourth, the `uv run` prefix in `mise.toml` makes the toolchain state observable from a single point — a developer who gets a wrong-Python error knows to check `mise install` against `.mise.toml`, not to chase a `$PATH` issue.

What was given up. Three costs are accepted. First, contributors can no longer edit a slash command by hand-editing a single file in the user's home directory; every change to a command's prompt body has to round-trip through a code edit in `src/deviate/prompts/commands/`, a `mise run test` cycle, and a `deviate setup` reinstall. Second, agents that want to *extend* DeviaTDD with their own commands cannot drop a file into `.claude/commands/` and expect it to be picked up by the framework — the framework only knows about commands it installed. Adding a project-local command is supported by the package's discover mechanism (`discover_commands()` in `src/deviate/core/commands.py`), but it is a different surface than the agent-platform mirror. Third, the Python-only runtime commits the project to a Python toolchain: contributors on systems without `uv` (or without access to PyPI) cannot run the framework's tasks at all. The accepted position is that this is the same audience already required for `mise install` and `uv sync`; relaxing it would re-introduce every dispatch ambiguity the redesign was meant to retire.

Rejected alternatives. Three were considered and turned down.

- **Bash wrappers that exec `python -m deviate.cli …`** — would have given the agent platforms a familiar shell entrypoint per command, at the cost of re-implementing layer routing in every wrapper, re-running the package-resolution step on each invocation, and accepting that the wrapper's Python interpreter is whatever is on `$PATH` rather than the `uv`-managed virtualenv. The previous shape embodied this alternative and was deprecated by ISS-001-005.
- **A separate top-level `prompts/` directory mirrored into the package via a sync script** — would have preserved the familiar "edit the Markdown file next to the code" workflow while still putting the resource inside the package, but would have introduced a sync step that could drift, a separate manifest of "what should be in `src/deviate/prompts/commands/`" that needed its own audit, and a category of bug ("file edited in `prompts/` but not yet synced") that the README would have to warn contributors about. Rejected because the asymmetry between the two locations was the bug.
- **A pluggable prompt backend (e.g., Jinja templates, an out-of-tree plugin system)** — would have given third-party contributors a way to ship custom slash commands without editing the package, but would have moved the source-of-truth boundary outside the lockfile. The `_LAYER_MAP` is small enough (14 lines) that the cost of a real plugin surface (a discovery mechanism, a versioning strategy, a security model) was not justified for the use case.

Each alternative would have made a single contributor experience easier; none would have preserved the unified prompt versioning or the in-process dispatch path. The python-only runtime is the cheapest design that keeps both.

## Implications

The python-only architecture shapes three concrete things about how the framework evolves. First, every change to a slash command's prompt body is a code change in the same PR as any matching code change: the package version, the `_LAYER_MAP`, and the prompt template all share a release. This collapses the class of "prompt-only fix" PRs the previous shape encouraged — those PRs were dangerous because they shipped behavioral changes against locked code. Second, the in-process dispatch path inside `deviate run` (`_PHASE_MAP` synchronous calls, `_run_pytest` subprocess only where unavoidable, no shell wrappers anywhere) means new micro-phase transitions can be added without introducing subprocess latency. The constitution budgets `L_max ≤ 200ms` per agent platform export during `deviate setup`, and that budget is reachable only because the install path is a pure-Python file copy and not a `cp` plus a permission fix plus a `$PATH` resolution. Third, agent platforms that want to participate must expose a slash-command discovery surface that maps onto one of the four supported directory conventions; a platform whose discovery convention is not file-based (e.g., a remote-only API) cannot be added without a parallel discovery layer in `core/commands.py`. The current `_get_agent_command_dir` enumerates the four supported shapes; adding a fifth is a code change, not a config change.

The architecture also constrains what cannot change. A future maintainer cannot move the prompt layer out of the Python package without also revisiting the version-pinning story, the `_LAYER_MAP` location, the `importlib.resources` contract in `assembly.py`, and the `_install_commands_to_agents` mirror behavior. A future maintainer who wanted to allow `.sh` command wrappers would have to redesign `discover_commands()` to know about two file extensions, redesign `install_command()` to validate the wrapper's `#!/usr/bin/env` shebang, redesign the mirror's gitignore pattern to whitelist `.sh` extensions, and document the failure mode for contributors who edit the wrapper but not the Markdown. None of those changes are impossible, but they are non-local: the python-only choice was made at the architectural level, and unwinding it touches every layer at once.

What becomes easier: a single toolchain (`mise` + `uv` + Python 3.13) covers development, testing, lint, format, and dispatch; a unified edit surface (`src/deviate/`) covers code and prompts; a single command (`deviate setup`) brings a fresh checkout into a runnable state. What becomes harder: contributors who want a fast iteration loop on prompts alone must wait for the full `mise run test` cycle; projects that want to ship custom commands live next to DeviaTDD must extend the package rather than drop files into the mirror.
