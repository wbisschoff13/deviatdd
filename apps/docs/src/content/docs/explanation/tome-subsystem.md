# How the Tome Subsystem Emerges

Why does DeviaTDD ship a seven-component subsystem whose entire job is to curate its own documentation, and why is that subsystem decomposed into one classifier, four per-quadrant writers, one verifier, and one setup scaffolder rather than one monolithic doc tool? The answer is a deliberate pairing of structural seams — quadrant discipline, frontmatter schema, and dependency gating — that, taken together, keep the documentation honest about what the codebase actually does without forcing contributors to maintain it by hand. The trade-offs that pairing accepts are about prompt-only iteration, the absence of machine-parseable handoffs, and a willingness to defer a CLI surface until usage data shows which phases need deterministic enforcement.

## Context

Documentation for a moving codebase is a hard problem because it has two masters. The first master is the code: every public surface — a CLI flag, a config field, a phase transition, a slash command — can change in a single commit, and the doc that described the old behavior becomes wrong the moment the commit lands. The second master is the reader: the same project serves a beginner who needs a learning narrative, an operator who needs to look up a flag, and a contributor who needs to understand why the system is shaped the way it is. These two masters pull in different directions, and a documentation system that serves them with one document type per topic tends to produce pages that are simultaneously too long, too short, and mis-targeted.

The taxonomy this codebase adopts is Diátaxis: four quadrants — `tutorial`, `how-to`, `reference`, `explanation` — each with its own register, its own audience, and its own purpose. The quadrants are not stylistic preferences — they are structural categories, and a page that mixes registers fails the taxonomy's promise to the reader. The problem the Tome subsystem solves is therefore not "how do we generate documentation?" but "how do we curate documentation that respects the taxonomy and stays in sync with the code without becoming the codebase's largest maintenance burden?"

The constraint that pushes the design is the constitution's tech-stack rule: no new Python runtime, no `package.json`, no Node toolchain in DeviaTDD's own repo. Tome must layer on top of the existing Python 3.13 + Typer stack using only the prompt layer. That constraint is the reason Tome v1 is prompt-only — every skill lives as a static `SKILL.md` markdown file under `src/deviate/prompts/skills/tome-*/`, and there is no `deviate tome <phase>` CLI surface in this iteration. The system that emerges is one of seven cooperating prompt skills, two read-only roles, and one structural seam that does the work most CLI tools would normally do: the quadrant directory.

## Rationale

Seven components is not an arbitrary count; it is the minimum number of roles that lets the system classify, write per quadrant, verify, and bootstrap itself without any one component doing two jobs. The Tome classifier (C1) ingests commit, branch, working-tree, or whole-codebase evidence and emits a classification report naming the required doc-type quadrants. Each of the four writers (C2 tutorial, C3 how-to, C4 reference, C5 explanation) consumes a capability row whose `doc_type` matches its quadrant and produces exactly one page under that quadrant's directory. The Tome verifier (C6) reads what the writers produced and runs cross-doc checks — factual consistency, path correctness, no cross-type contamination, valid Starlight location — without writing anything back. The Tome setup (C7) is the idempotent bootstrap that scaffolds `apps/docs/`, the four quadrant directories, and the Starlight `content.config.ts` schema extension; it is the only component that touches `apps/docs/` outside a quadrant subdirectory. Collapsing any two of these roles into one component would have blurred the gate between "decide what to document" and "write the document" and made per-writer self-verify impossible to test in isolation.

The quadrant discipline is the structural seam that holds the system together. Every writer is confined by a hard path rule: `tome-write-tutorial` writes only to `tutorials/`, `tome-write-how-to` writes only to `how-to/`, `tome-write-reference` writes only to `reference/`, and `tome-write-explanation` writes only to `explanation/`. A writer that needs to touch a different quadrant emits a `[REJECT]` boundary violation and halts; auto-routing to another writer is forbidden. The directory itself is the boundary — there is no policy layer above it that could be bypassed by a `--force` flag. The benefit is that cross-type contamination (a how-to accidentally containing tutorial narrative) is structurally impossible at write time: the writer whose register would produce the contamination does not own the directory it would land in. The cost is that cross-quadrant cleanup always requires re-classification through C1.

The dependency graph is the third design choice and it is the simplest. C7 is the only prerequisite for C1: until `apps/docs/` exists, C1 emits `setup-required` and halts. C1 is the only prerequisite for C2-C5: a writer fires only when C1's capability table contains a row whose `doc_type` matches the writer's quadrant and whose `action` is `create` or `update`. C6 runs only after at least one writer has produced a file; C2-C5 are independent of each other, so the verifier does not wait for the whole set. The result is a strictly acyclic flow: C7 → C1 → {C2, C3, C4, C5} → C6, with no back edges, no retries, and no automatic re-runs. If the verifier fails, a human reads the report and manually re-runs the relevant writer with updated evidence — there is no `<judge_feedback>` auto-routing and no automated retry loop.

The frontmatter schema is the final design choice and it is duplicated on purpose. Every page emitted by C2-C5 carries seven YAML fields — `title`, `description`, `doc_type`, `status`, `last_verified_at`, `verified_sha`, `related_issues` — and that schema is declared in two places: the Starlight-side `content.config.ts` (which extends `docsSchema()`) and inline in each writer's prompt (which the LLM consults before emitting). The duplication is the cost of running on top of Starlight: the LLM-side schema has to be self-contained because there is no machine-readable schema file the writer could import. The benefit is that the LLM cannot forget a field or emit a wrong type without the verifier catching it; C6 checks `doc_type`-vs-path agreement and the seven-field presence as part of its cross-doc pass. The two declarations must agree — drift between them is a verifier finding.

## Mental Model

Picture the Tome subsystem as a small post office. C7 is the building — it has to exist before any letter can be processed. C1 is the sorting desk: a commit or a branch diff arrives, C1 reads the evidence and emits a sorting slip (the classification report) that says which quadrant each capability belongs to and where the page should land. The four writers (C2-C5) are four mail carriers, each responsible for exactly one route. A carrier cannot deliver mail to another carrier's route — the post office is structured so that the constraint is enforced by the building's doors, not by the carriers' promises. C6 is the auditor who walks the building after the carriers have finished and reports whether each letter reached the right route, whether each envelope carries the right stamps, and whether the contents match the address.

```

        exists?  ▼                          │ reports
              ┌──────┐                      │

                 │                          │
       ┌─────┬───┴────┬─────┐               │

     │C2 │ │C3 │   │C4 │ │C5 │ ────────────┘

     FLOW-05 FLOW-06 FLOW-07 FLOW-08
```

The seven flows (FLOW-04 through FLOW-10) are the traceable edges of this diagram. The classifier-to-writer edges are *per-row*: a writer fires only when C1's report contains a row for its quadrant. The writer-to-verifier edges are *cumulative*: C6 reads whatever the writers produced and does not care which writer produced what. There is no edge from C6 back to a writer — the auditor's report is the end of the line for this commit, and any remediation happens in the next commit's classification pass.

## Trade-Offs

The Tome subsystem earns three properties the codebase values: structural quadrant purity (a page in `explanation/` cannot contain operator steps because no writer that could produce operator steps owns that directory), zero-runtime curation (no Python module, no CLI sub-app, no Node toolchain is added to DeviaTDD's repo to make docs happen), and idempotent bootstrap (C7 re-runs produce zero diff, so a maintainer can run it on every release without coordination). It also gives up four things, and being honest about them is part of why the design is the shape it is.

What was given up. First, there is no machine-parseable handoff between components. C1 emits markdown that a human reads; the human then invokes the relevant writer manually. There is no `tome/contracts.py` shared schema, no structured metadata file, no automatic dispatch. The benefit is that the system requires no runtime; the cost is that the classification-to-write step is human-mediated and a careless operator can drop a capability row before invoking the writer. Second, there is no automatic re-routing from the verifier. C6 emits a human-readable report; if a page fails cross-doc checks, a human reads the report, identifies the failing writer, and re-runs it with updated evidence. The benefit is that the verifier does not need a writable channel back into the system; the cost is that multi-page failures across multiple writers require a human to drive each re-run by hand. Third, there is no CLI surface for the Tome phases in v1. There is no `deviate tome <phase>` sub-app, no machine-enforced path validation, no deterministic frontmatter lint. The benefit is that v1 ships pure-prompt and gathers usage data first; the cost is that every Tome phase currently relies on the agent platform's slash-command infrastructure. Fourth, the frontmatter schema is declared in two places — Starlight-side `content.config.ts` and inline in each writer prompt — and drift between them is only caught at verify time.

Rejected alternatives. Four were considered and turned down.

- **A unified writer that handled all four doc types** would have collapsed the seven-component shape into three components (classifier, writer, verifier) and would have made the system simpler to invoke, but it would have lost the structural quadrant seam. The boundary between "this page belongs to `how-to/`" and "this page belongs to `reference/`" would have become a policy check inside one writer rather than a directory-level rule enforced by the filesystem. Rejected because the value of quadrant discipline is the impossibility of cross-type contamination, and that impossibility depends on the writer owning the directory.
- **A `deviate tome <phase>` CLI surface in v1** would have given every Tome phase a programmatic entry point with deterministic path validation, machine-enforced frontmatter lint, and exit-code semantics. It was deferred to a future iteration once v1 usage data shows which phases actually need deterministic enforcement; committing to one shape before usage data is in hand would have locked in assumptions the data might not support. Rejected because the prompt-only v1 is cheaper to ship and easier to revise.
- **An auto-routing JUDGE pattern** — a structured `<judge_feedback>` payload that the verifier emits and the relevant writer consumes, with automatic re-runs until convergence — would have removed the human from the remediation loop. It was rejected because the verifier is intentionally read-only (its output is a report a human reads, not a feedback file a writer consumes) and because auto-routing would have made the verification step itself a component that can change the system, breaking the read-only invariant.
- **A centralized `tome/contracts.py` shared schema module** would have given C1, C2-C5, and C6 a single Python source of truth for the capability row schema, the frontmatter schema, and the action enum. It was rejected because v1 is prompt-only — there is no `tome/` Python module — and because the inline-schemas-in-prompts approach lets the LLM consult a single self-contained block at write time without resolving an external module. The trade-off is that schema drift between the two places is possible and must be caught by the verifier.

Each alternative would have made a single contributor experience easier; none would have preserved the structural quadrant seam, the prompt-only v1, or the read-only verifier. The seven-component shape is the smallest design that keeps all three.

## Implications

The Tome subsystem shapes five concrete things about how documentation evolves in this project. First, every change to a documentation surface flows through C1's evidence-gathering pass; the classifier is the entry point that decides whether a commit warrants a `tutorial`, `how-to`, `reference`, or `explanation` page. The implication is that adding a new doc type requires a new writer component — the taxonomy is closed at four quadrants by design. Second, every writer's emit is gated by a self-verify pass — quadrant path, frontmatter schema, register discipline, `See Also` presence — and that pass is built into the writer prompt, not into a separate CI step. A writer that emits a page failing its own self-verify halts before delivery. Third, the prompt-only constraint means every Tome phase is reachable only through the agent platform's slash-command infrastructure. A maintainer who wants to run C1 outside an agent session must invoke the prompt manually or use a generic LLM client that loads the skill file. Fourth, the duplicated frontmatter schema means a Starlight-side schema change (a new field in `content.config.ts`) must be mirrored into every writer prompt in the same commit; the verifier catches the drift, but the cost is paid in the verify pass. Fifth, the verifier's read-only stance means cross-quadrant cleanup is a multi-writer coordination problem: if a page in `reference/` contains tutorial content, the fix is to re-classify the capability, re-run `tome-write-tutorial`, and have the new tutorial page supersede the contaminated reference page — the verifier cannot rewrite the reference page in place.

What becomes easier: a contributor can ship a documentation change by writing one `SKILL.md` prompt change and one source-tree change in the same PR; a maintainer can audit the documentation surface by reading the seven `tome-*/SKILL.md` files; the docs system itself is described in the same `tome-write.md` reference surface that describes every other writer. What becomes harder: cross-quadrant consistency checks (only possible at the verifier's cross-doc pass), automated documentation pipelines (no auto-routing), schema evolution (two-place declaration must be kept in sync), and a CLI-driven workflow for teams that prefer deterministic tooling over prompt-driven flows.

The architecture also constrains what cannot change. A future iteration cannot collapse the seven components into fewer without revisiting the quadrant discipline, the dependency graph, and the read-only verifier invariant. A future iteration cannot remove the duplicated frontmatter schema without also revisiting the prompt-only v1 constraint or the Starlight-side `docsSchema()` extension. A future iteration cannot add a fifth doc type without adding a new component, a new quadrant directory, and a new writer prompt. None of those changes are impossible, but they are non-local: the seven-component shape was made at the architectural level, and unwinding it touches every writer, the verifier, and the classifier at once.
